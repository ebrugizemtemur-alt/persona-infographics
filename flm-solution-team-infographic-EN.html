⚠️ PROBLEMATIC SCENARIO
Flight at 22:00 UTC delayed to 01:00 UTC next day. NetLine sends TIM update (same UTC day), but COMMIS converts to local time → sees day change → creates NEW record instead of update. Result: Duplicate.
✅ FUTURE STATE (NetLine++)
Both leg-based AND flight-based definitions supported. JSON messages via Kafka. Turkish Technology building middleware to convert to XML for COMMIS compatibility.
📨 MESSAGE TYPES
SSM (Standard Schedule Message)
Real-time flight changes. Types: NEW, TIM, CNL. Pushed from NetLine via telex format.
SSIM (Standard Schedules Info Message)
Full season schedule. Uploaded manually at season start (summer/winter).
FLTREQUEST (XML)
Rich flight data including pax count, fuel. Non-standard, but needed for capacity estimation.
FLTSEGCAPUTL
COMMIS → NetLine. Capacity utilization, AVI/HUM flags. Only reverse flow message.
"SSM content was not rich enough (pax count, fuel), so FLTREQUEST was created. Ideally each field should be a separate service."
— Solution Team Meeting
